# api-plus 上游渠道安全巡检方案（V1）

> 日期：2026-03-22  
> 适用项目：api-plus  
> 阶段：设计方案 / 待实现  
> 范围：**仅接入定时渠道巡检，不拦截线上真实用户请求**

---

## 1. 一句话结论

api-plus 需要在现有“渠道存活检测 / 自动巡检”链路中，增加一层 **上游返回内容安全检测**：

> **渠道巡检成功后，对返回文本内容做恶意 payload 扫描；命中高危规则时，将渠道状态切换为 `security_disabled`，且该状态不得被后续自动巡检恢复。**

这不是锦上添花，而是 api-plus 作为网关产品必须补上的 **供应链安全底座**。

---

## 2. 背景与问题定义

当前 api-plus 已有：

- 渠道可用性检测
- 自动禁用/自动恢复
- 响应时间阈值禁用
- 多 Key 渠道状态管理

但现有检测只关心：

- 渠道活不活
- 返回是否报错
- 响应是否太慢

**没有处理一种更危险的情况：**

上游渠道虽然“可用”，但在返回内容中故意混入恶意内容，例如：

- prompt injection 指令
- HTML / JS / markdown 载荷
- 命令执行诱导片段
- 可疑 callback / base64 / 混淆片段
- 为后续前端渲染、Agent 工作流或人工复制粘贴埋下后门

这类问题的危险点在于：

1. 渠道健康检测会误判为“正常”
2. 自动恢复逻辑可能把已发现风险的渠道重新打开
3. 风险会沿网关向下游扩散，污染你的系统和用户

所以必须把“**活着**”和“**安全**”分开判断。

---

## 3. 本次 V1 目标

V1 的目标不是做一个通用安全平台，而是先做一个 **渠道安全熔断层**。

### 3.1 要达成的目标

1. 在定时巡检中检测上游返回文本内容是否含恶意 payload
2. 引入独立状态 `security_disabled`
3. 命中高危规则后自动切换为 `security_disabled`
4. `security_disabled` **绝不允许自动恢复**
5. 保存命中原因、规则、时间、样本证据
6. 后台能明确展示“这是安全禁用，不是普通禁用”

### 3.2 明确不做

本阶段不做：

- 线上真实用户请求实时拦截
- HTTP 响应头 / redirect / content-type 联动检测
- LLM 二次判定 / 小模型安全分类器
- 独立安全审计中心
- 自动修复和自动解封

原因很简单：

> 当前最优先的是把“**安全问题不会被当成普通故障处理**”这件事先做对。

---

## 4. 设计原则

### 原则 1：安全禁用和普通禁用彻底分开

普通自动禁用（例如额度不足、上游 403、响应超时）和安全禁用不是一回事。

如果继续复用 `auto_disabled`，会出现：

- 定时任务发现恶意内容 → 自动禁用
- 下一轮存活检测发现接口可通 → 自动恢复

这相当于系统自己把门重新打开。

### 原则 2：安全状态只允许人工恢复

`security_disabled` 一旦触发：

- 不得自动恢复
- 不得被“渠道存活”覆盖
- 必须由人工检查后手动恢复

### 原则 3：先做规则检测，别先上复杂智能判定

首版先用规则扫描，优先解决 80% 明显风险：

- 可解释
- 成本低
- 易于调试
- 不会引入新的模型依赖和误报复杂度

### 原则 4：检测结果必须可追溯

不能只给一个“命中风险”。

至少要记录：

- 命中哪条规则
- 当时返回了什么
- 何时发生
- 对渠道做了什么动作

不然以后排查时会非常痛苦。

## 5. 状态机设计

## 5.1 现有状态

当前已有：

- `ChannelStatusEnabled = 1`
- `ChannelStatusManuallyDisabled = 2`
- `ChannelStatusAutoDisabled = 3`

## 5.2 新增状态

新增：

- `ChannelStatusSecurityDisabled = 4`

## 5.3 语义定义

| 状态 | 含义 | 是否自动恢复 |
|---|---|---|
| enabled | 正常可用 | 不适用 |
| manually_disabled | 人工禁用 | 否 |
| auto_disabled | 可用性/额度/普通异常导致自动禁用 | 可按现有逻辑恢复 |
| security_disabled | 安全巡检命中高危风险后强制禁用 | **否，只能人工恢复** |

## 5.4 恢复规则

### 允许自动恢复的状态

- 仅 `auto_disabled`

### 禁止自动恢复的状态

- `security_disabled`
- `manually_disabled`

---

## 6. 巡检流程设计

V1 仅接入现有 `testAllChannels()` 定时巡检流程。

## 6.1 当前流程（简化）

1. 遍历渠道
2. 调用 `testChannel()`
3. 判断是否需要自动禁用
4. 对普通自动禁用渠道尝试自动恢复
5. 更新响应时间

## 6.2 新流程

改造后流程：

1. 遍历渠道
2. 若状态为 `manually_disabled`，跳过
3. 调用 `testChannel()`
4. 若 `testChannel()` 成功，获取测试响应文本
5. 对响应文本做安全扫描
6. 若命中高危规则：
   - 将渠道设为 `security_disabled`
   - 记录安全证据
   - 跳过自动恢复逻辑
7. 若未命中安全风险：
   - 继续按现有逻辑处理普通自动禁用 / 自动恢复
8. 更新响应时间

## 6.3 关键约束

- `security_disabled` 渠道下一轮巡检即使“响应正常”也**不得自动恢复**
- 自动恢复逻辑必须明确排除 `security_disabled`

---

## 7. 安全检测范围（V1）

V1 只扫描：

> **上游返回的文本内容**

包括：

- 普通 JSON 响应 body
- SSE/流式响应的文本片段
- markdown / text / message 内容

不包括：

- HTTP header
- redirect
- content-type
- 前端 DOM 渲染结果
- 下游执行链

---

## 8. 检测规则设计

## 8.1 设计思路

参考 OpenGuardrails 的思路，首版采用：

> **策略化规则扫描 + 风险分级 + 自动动作**

先做规则，不急着接模型分类器。

## 8.2 检测类别

### A. Prompt Injection 关键词

用于识别试图污染后续 Agent / 系统提示链路的内容，例如：

- “忽略之前所有指令”
- “泄露系统提示词”
- “执行以下命令”
- “请访问这个链接继续完成任务”

### B. HTML / JS / 富文本注入片段

例如：

- `<script`
- `javascript:`
- `iframe`
- `onload=`
- `onerror=`
- `svg`

### C. 命令执行诱导片段

例如：

- `curl`
- `wget`
- `bash -c`
- `powershell`
- `rm -rf`
- `chmod +x`

### D. 可疑混淆载荷

例如：

- 超长 base64 片段
- 大量十六进制转义
- 可疑 data URI

### E. 可疑外链 / callback 诱导

例如：

- 明显非模型回复语义、却指向可疑外部地址
- 疑似控制平面回连地址

## 8.3 风险分级

V1 先简化为两档：

- `safe`：未命中高危规则
- `high_risk`：命中高危规则，立即熔断

也就是说，V1 先不做复杂的 `warning` 自动流转，先把高危风险挡住。

---

## 9. 数据模型设计

## 9.1 状态字段

继续复用 `channels.status`，新增一个合法值：

- `4 = security_disabled`

## 9.2 证据存储

为了降低首版改动成本，V1 先复用 `Channel.OtherInfo`。

建议新增这些键：

| Key | 含义 |
|---|---|
| `status_reason` | 当前状态原因 |
| `status_time` | 当前状态时间 |
| `security_reason` | 安全命中摘要 |
| `security_rule_ids` | 命中规则 ID 列表 |
| `security_detected_at` | 命中时间 |
| `security_sample` | 截断后的命中样本 |
| `security_source` | 触发来源，固定为 `channel_auto_test` |

## 9.3 样本存储原则

不要全量无限保存原始返回体。

建议：

- 最多保留前 `N` 个字符（如 1KB~2KB）
- 必要时脱敏
- 只保留足够复盘的样本

避免把日志系统变成新的风险源。

## 10. 后端实现设计

## 10.1 新增模块建议

建议新增一个独立安全检测模块，例如：

- `service/channel_security.go`
或
- `common/channel_security.go`

职责：

1. 定义规则
2. 扫描文本
3. 返回结构化结果

建议返回结构：

```go
type ChannelSecurityScanResult struct {
    Safe        bool
    RiskLevel   string
    Reason      string
    RuleIDs     []string
    Sample      string
}
```

## 10.2 testResult 扩展

当前 `testChannel()` 虽然会读取响应 body 并打印日志，但返回结构里没有携带响应文本。

建议给 `testResult` 增加：

- `responseBody []byte`

这样巡检层可以直接使用测试结果做安全扫描，而不用重复请求。

## 10.3 自动禁用入口

建议新增一个专门动作函数，例如：

- `service.SecurityDisableChannel(...)`

用途：

- 设置 `status = security_disabled`
- 写入 `OtherInfo`
- 发通知
- 与普通 `DisableChannel` 区分

不要把安全禁用塞进普通 `DisableChannel` 里混着处理，不然后面一定越改越乱。

## 10.4 自动恢复保护

当前 `ShouldEnableChannel()` 只允许 `auto_disabled` 自动恢复。

这个函数要继续保持这个原则，且测试明确覆盖：

- `security_disabled` 不自动恢复

---

## 11. 前端展示设计

后台渠道列表和相关状态展示要支持新状态：

- `security_disabled` → 展示文案：**安全禁用**

建议：

- 颜色：红色 / 深橙
- tooltip / 详情中显示：
  - 安全原因
  - 命中规则
  - 检测时间

目标很简单：

> 管理端一眼能看出这个渠道不是“挂了”，而是“脏了”。

---

## 12. 测试设计

本功能必须按 TDD 做，至少覆盖以下行为。

## 12.1 单元测试

### 安全扫描器

1. 命中 `<script>` 时返回高危
2. 命中 prompt injection 关键词时返回高危
3. 普通正常文本不命中
4. 样本会被截断

### 状态机

1. `security_disabled` 不会被 `ShouldEnableChannel()` 自动恢复
2. `auto_disabled` 仍可按原逻辑自动恢复

## 12.2 巡检流程测试

1. `testChannel()` 成功且响应命中高危规则  
   → 渠道进入 `security_disabled`

2. `security_disabled` 渠道下一次巡检即使正常  
   → 仍保持 `security_disabled`

3. 普通失败渠道  
   → 仍按原有 `auto_disabled` 逻辑处理

## 13. 实施顺序

建议按这个顺序落地：

### 第一步：数据与状态机

- 增加 `ChannelStatusSecurityDisabled`
- 前后端能识别这个状态

### 第二步：安全扫描器

- 独立实现规则检测
- 先完成单元测试

### 第三步：巡检接入

- 扩展 `testResult` 携带响应文本
- 在 `testAllChannels()` 中接入扫描

### 第四步：熔断动作

- 增加 `SecurityDisableChannel`
- 写通知、写 `OtherInfo`

### 第五步：展示与回归

- 后台展示“安全禁用”
- 验证不会误自动恢复

---

## 14. 风险与已知局限

### 风险 1：规则误报

有些正常文本可能会包含：

- “script”
- “curl”
- “javascript”

所以规则不能只做极粗暴关键词匹配，建议用：

- 大小写归一化
- 组合命中
- 部分场景正则

### 风险 2：规则漏报

更隐蔽的语义型攻击可能绕过首版规则。

这是可接受的，因为 V1 目标是先挡住明显高危载荷，而不是一步做到完美。

### 风险 3：证据泄漏

如果把原始返回体完整存库，可能把风险样本长期留在系统里。

所以一定要：

- 截断
- 脱敏
- 控制展示范围

---

## 15. 最终建议

这件事应该作为 api-plus 的 **P0 安全能力** 来做。

当前最合适的落地方式不是“加一个安全优化项”，而是：

> **把它纳入渠道巡检主流程，作为独立状态机的一部分。**

最关键的不是“检测出多少花活”，而是先保证：

1. 危险渠道能被识别  
2. 被识别后能被强制熔断  
3. 熔断后不会被系统自动放出来

这三件事先做成，api-plus 的渠道治理才算真正进入可控状态。
